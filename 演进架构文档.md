## 1. 演进历程

### 1.1 第一阶段：长期记忆插件（初心）

**设计初衷**：

> "我想做一个记忆系统作为长期记忆插件，在 AstrBot 这个聊天机器人中使用"

**核心需求**：

- 记录用户与 AI 的交互历史
- 支持长期记忆的存储与检索
- 解决传统聊天机器人"聊完就忘"的问题

**技术实现**：

```python
# core/memory_manager.py - 记忆管理核心

    """主动式记忆管理系统"""
    
    # 日记系统 - 按日期存储 Markdown 文件
    def record_interaction(self, uid: str, group_id: str, role: str, content: str):
        """记录交互到日记"""
        # 实现：写入 {data_dir}/{uid}/memory/{YYYY-MM-DD}.md
    
    # 记忆提取 - 关键词触发
    async def extract_memories_from_text(self, text: str):
        """从文本中提取记忆"""
        # 实现：检测"记住、别忘了、重要"等关键词
```

**文件结构**：

```
{data_dir}/
└── {uid}/
    ├── SOUL.md           # 核心人格
    ├── MEMORY.md         # 长期记忆
    └── memory/
        ├── 2026-03-10.md  # 日记
        └── 2026-03-11.md
```

### 1.4 第四阶段：多用户管家（深化）

**需求演进**：

> "群的观念既然引入了，就想到群里面有很多群成员，他们在和这个机器人聊天的时候，这个机器人也能分身成他们的个人管家，就引入了多用户的概念。"

**核心能力**：

- 群内每个成员都有独立的个人画像
- 机器人能识别不同群成员的身份
- 回复时注入该成员的个人画像 + 群体画像 + 相关记忆

**技术实现**：

### 1.5 第五阶段：消息优化（体验提升）

**需求演进**：

> "再后来就像用户初次唤醒机器人进行引导一样，我们在群中的唤醒也给他了一个群引导的版本。至此，聊天功能大概我能想到的就是这样了。"

**优化点**：

1. **分段回复**：QQ/微信不支持 Markdown，需要清洗格式并分段
2. **引用回复**：增强对话连贯性
3. **@提及**：精准触达群成员
4. **免@唤醒**：更像真人
5. **连续对话**：多轮对话智能判定

**技术实现**：

#### 1.5.1 智能分段发送器

```python
# core/smart_sender.py
class SmartSender:
    """智能分段发送器"""
    
    def split_text(self, text: str) -> List[Segment]:
        """
        智能分段：
        1. 正则表达式分段（按句号、问号、感叹号）
        2. 保护社交组件（@提及不被切断）
        3. Markdown 表格保护
        """
        
    async def send_with_typing_delay(self, segments: List[Segment]):
        """
        拟人化打字延迟：
        - 基于字符数计算延迟
        - 添加随机波动因子
        - 会话级并发锁（防止消息穿插）
        """
```

## 3. 记忆系统详解

### 3.1 多层次记忆体系

```
热记忆 (Hot Memory)
└─→ 当前会话的即时上下文 (对话总账 SQLite)
  
冷记忆 (Cold Memory)  
└─→ 基于日期的 Markdown 日记 (按天存储)
  
归档记忆 (Archived Memory)
└─→ 经过"睡眠巩固"提炼的核心事实 (MEMORY.md)
```

### 3.2 记忆记录流程

```python
# mixins/events_mixin.py
async def _process_single_message(self, uid, group_id, session_id, user_name, message, mentions):
    """处理单条消息"""
    
    # 1. 记录到对话总账（热记忆）
    await self.conversation_ledger.add_message(
        session_id=session_id,
        role="user",
        content=message,
        source="user_input"
    )
    
    # 2. 记录交互到日记（冷记忆）
    await self.memory_manager.record_interaction(
        uid, group_id, user_name, message
    )
    
    # 3. 检测跨群消息
    await self.cross_group_system.detect_and_route(
        group_id, message, uid, user_name
    )
```

```
LLM 生成针对性回复：
"哎呀，感冒了要多休息呀！记得你之前说过对某些药过敏，
不要吃含 XX 成分的感冒药。群里大家也很关心你呢~"
```

```python
# mixins/events_mixin.py
@filter.on_llm_request()
async def before_llm_request(self, event: AstrMessageEvent, req: ProviderRequest):
```

```python
    引用决策：
    
    1. AI 决定 target_msg_id（引用哪条消息）
    2. 完全尊重 AI 的决策
    3. 如果 AI 返回 null，表示综合回复，不引用任何消息
    """
    
    decision = await self._ai_decide_reply_target(...)
    
    if decision.should_reply:
        result.chain = [Plain(decision.reply_text)]
        
        if decision.target_msg_id and self.config.smart_split_group_reply:
            result.chain.insert(0, Reply(id=decision.target_msg_id))
```

### 11.3 隐私边界与跨场景搜索

**核心问题**：

> "在聊天的时候我们还要解决一些边界和隐私的问题。在群内聊天的时候，我们就不希望他能够把我们私人的事情抖出来。在私聊的时候，我们又希望能够获取所有的和我们有关的信息。需要跨群。"

**解决方案**：三级隐私边界机制

***

#### 11.3.1 搜索范围定义

| Scope        | 群聊场景             | 私聊场景  | 需要理由   |
| ------------ | ---------------- | ----- | ------ |
| **group**    | 仅当前群聊            | ❌ 不可用 | ❌      |
| **personal** | 当前群聊 + 私聊        | ❌ 不可用 | ✅ 群聊需要 |
| **cross**    | 当前群聊 + 私聊 + 所有群聊 | 默认使用  | ✅ 群聊需要 |

#### 11.3.2 记忆搜索实现

```python
# mixins/tools_mixin.py: memory_search
@filter.llm_tool()
async def memory_search(
    self,
    event: AstrMessageEvent,
    query: str,
    scope: str = "group",  # 默认仅当前群聊
    limit: int = 5,
    cross_reason: str = ""  # 跨场景搜索理由
):
    """
    检索记忆系统中的相关内容。
    
    Args:
        scope (str): 搜索范围。
            - group: 仅搜索当前群聊的群体记忆（群聊默认，私聊不可用）
            - personal: 当前群聊 + 私聊记忆（群聊需提供理由，私聊不可用）
            - cross: 当前群聊 + 私聊 + 所有其他群聊（群聊需提供理由，私聊默认）
        cross_reason (str): 【重要】跨场景搜索理由。
            群聊场景下使用 personal 或 cross 时必填。
            私聊场景不需要此参数。
    """
    
    uid, group_id, _ = self._get_identity(event)
    is_private_context = group_id == "private"
    
    # 私聊场景：自动使用 cross 范围
    if is_private_context:
        scope = "cross"
    
    # 群聊场景：跨范围搜索需要理由
    if not is_private_context and scope in ("personal", "cross"):
        if not cross_reason or not cross_reason.strip():
            return (
                f"❌ 跨场景搜索必须提供 cross_reason 参数。\n"
                f"请在 cross_reason 参数中引用用户的话，说明为什么需要跨场景搜索。"
            )
    
    # 执行搜索
    if scope == "group":
        results = await self.search_engine.search(
            query, uid, group_id, "group", limit
        )
    
    elif scope == "personal":
        # 群聊 + 私聊
        if not is_private_context:
            results = await self.search_engine.search(
                query, uid, group_id, "group", limit // 2
            )
        private_results = await self.search_engine.search(
            query, uid, "private", "personal", limit // 2
        )
        results.extend(private_results)
    
    elif scope == "cross":
        # 群聊 + 私聊 + 所有群聊
        results = await self.search_engine.search(
            query, uid, group_id, "cross", limit
        )
```

```python
# mixins/media_tools_mixin.py: search_media
@filter.llm_tool()
async def search_media(
    self,
    event: AstrMessageEvent,
    query: str = "",
    scope: str = "group",
    media_type: str = "all",
    days: int = 30,
    limit: int = 10,
    cross_reason: str = ""
):
    """
    搜索媒体库中的图片和文件。
    
    隐私分级（与记忆搜索一致）：
    - group: 仅当前群聊
    - personal: 当前群聊 + 私聊
    - cross: 当前群聊 + 私聊 + 所有其他群聊
    """
    
    uid, group_id, _ = self._get_identity(event)
    is_private_context = group_id == "private"
    
    # 私聊场景：自动使用 cross 范围
    if is_private_context:
        scope = "cross"
    
    # 群聊场景：跨范围搜索需要理由
    if not is_private_context and scope in ("personal", "cross"):
        if not cross_reason:
            return "❌ 跨场景搜索必须提供 cross_reason 参数"
    
    all_results = []
    
    if scope == "group":
        # 仅搜索当前群聊
        results = await self.media_manager.search_media(
            uid, group_id, query, media_type, days, limit
        )
        all_results.extend(results)
    
    elif scope == "personal":
        # 当前群聊 + 私聊
        if not is_private_context:
            results = await self.media_manager.search_media(
                uid, group_id, query, media_type, days, limit
            )
            all_results.extend(results)
        
        private_results = await self.media_manager.search_media(
            uid, "private", query, media_type, days, limit
        )
        for r in private_results:
            r["source"] = "私聊"
        all_results.extend(private_results)
    
    elif scope == "cross":
        # 当前群聊 + 私聊 + 所有群聊
        results = await self.media_manager.search_media(
            uid, group_id, query, media_type, days, limit
        )
        all_results.extend(results)
        
        # 私聊媒体库
        private_results = await self.media_manager.search_media(
            uid, "private", query, media_type, days, limit
        )
        for r in private_results:
            r["source"] = "私聊"
        all_results.extend(private_results)
        
        # 其他群聊媒体库
        joined_groups = self.group_manager.get_user_joined_groups(uid)
        for gid in joined_groups:
            if gid == group_id:
                continue
            group_results = await self.media_manager.search_media(
                uid, gid, query, media_type, days, limit // len(joined_groups)
            )
            for r in group_results:
                r["source"] = f"群聊:{gid}"
            all_results.extend(group_results)
```

```python
# mixins/tools_mixin.py: import_file_to_archive
@filter.llm_tool()
async def import_file_to_archive(
    self,
    event: AstrMessageEvent,
    filename: str,
    display_name: str = None,
    description: str = ""
):
    """
    将媒体库中的文件导入到档案馆（支持 Excel、CSV、TXT）。
    
    【重要】仅在用户明确要求"导入文件到档案馆"时才调用。
    用户仅发送文件而未说明要导入时，应先询问，不要自动调用。
    
    导入规则：
    - 日常模式 + 私聊 → 导入到个人档案馆
    - 日常模式 + 群聊 → 导入到群组档案馆
    - 管理员模式 (sudo) → 导入到全局档案馆
    """
    
    uid, group_id, _ = self._get_identity(event)
    is_sudo = self.identity_manager.is_sudo(uid, self.config.admin_uids)
    
    # 1. 获取文件路径
    file_path = self.media_manager.get_file_path(uid, group_id, filename)
    
    # 2. 验证文件类型
    ext = file_path.suffix.lower()
    supported_exts = ['.xlsx', '.xls', '.csv', '.txt']
    
    if ext not in supported_exts:
        return f"❌ 不支持的文件类型：{ext}"
    
    # 3. 确定档案馆路径（个人/群体/全局）
    router = ArchiveRouter(self.data_dir)
    db_path, scope = router.resolve_db_path(uid, group_id, is_sudo)
    
    # 4. 导入到档案馆
    manager = ArchiveManager(str(db_path))
    ingestor = DataIngestor(archive_manager=manager)
    
    table_name, row_count = ingestor.ingest_excel(
        file_path=str(file_path),
        display_name=display_name or filename,
        description=description
    )
    
    return (
        f"✅ 文件导入成功！\n"
        f"   原始文件：{filename}\n"
        f"   目标位置：{scope_label}\n"
        f"   表名：{table_name}\n"
        f"   行数：{row_count}\n\n"
        f"现在可以使用 `query_archives` 工具查询这个表的数据。"
    )
```

### 12.4 记忆系统的渐进式披露

#### 12.4.1 问题背景

**挑战**：

> "在记忆系统中也引入了渐进式披露。目的是继续提高我们上下文的性价比"

**问题**：

- PROFILE.md、MEMORY.md、SOP.md 等文件体积大
- 全量加载占用大量 Token
- AI 不需要一次性读取所有记忆

#### 12.4.2 渐进式披露架构

**核心思想**：

```
1. 一切皆节点：无论是 PROFILE.md 还是 skills/xxx/SKILL.md
2. 目录即 Prompt：System Prompt 只包含精简的文件目录树
3. 按需读取：AI 通过 file_read_tool 主动获取需要的节点内容
4. 节约 Token：基础 Prompt 从几千 Token 降至几百 Token
```

#### 12.4.3 ContextIndexer 实现

```python
def _extract_frontmatter_summary(self, file_path: Path) -> Optional[str]:
    """从文件的 YAML Front Matter 中提取 summary 字段"""
    content = file_path.read_text(encoding="utf-8")
    if content.startswith("---"):
        parts = content.split("---", 2)
        if len(parts) >= 3:
            summary_match = re.search(r'summary:\s*["\']?(.+?)["\']?\s*$', 
                                      parts[1], re.MULTILINE)
            if summary_match:
                return summary_match.group(1).strip()
    return None
```

```python
def _hash_password(password: str) -> str:
    """使用 bcrypt 对密码进行哈希"""
    import bcrypt
    salt = bcrypt.gensalt()
    hashed = bcrypt.hashpw(password.encode('utf-8'), salt)
    return hashed.decode('utf-8')

def _verify_password(password: str, hashed: str) -> bool:
    """验证密码是否匹配哈希值"""
    import bcrypt
    try:
        result = bcrypt.checkpw(password.encode('utf-8'), hashed.encode('utf-8'))
        return result
    except Exception as e:
        logger.warning(f"[Scriptor] 密码验证异常：{e}")
```

**2. CSRF 保护**：

```python
class CSRFManager:
    """CSRF 保护管理器"""

    _tokens: dict = {}

    @classmethod
    def create_token(cls, session_id: str) -> str:
        """为会话创建 CSRF 令牌"""
        token = generate_csrf_token(session_id)
        cls._tokens[session_id] = {
            "token": token,
            "created_at": datetime.now()
        }
        return token

    @classmethod
    def verify(cls, session_id: str, token: str) -> bool:
        """验证 CSRF 令牌"""
        if session_id not in cls._tokens:
            return verify_csrf_token(session_id, token)

        stored = cls._tokens[session_id]
        if (datetime.now() - stored["created_at"]).total_seconds() > CSRF_TOKEN_EXPIRE_MINUTES * 60:
            del cls._tokens[session_id]
            return False

        return token == stored["token"] and verify_csrf_token(session_id, token)
```

#### 15.3.4 任务触发

```python
def check_and_trigger(self, callback) -> List[ScheduledTask]:
    """检查并触发到期任务"""
    current_time = time.time()
    triggered_tasks = []

    with self._lock:
        for task in self.tasks:
            if not task.is_active:
                continue

            should_trigger = False

            if task.task_type == "once":
                if current_time >= task.trigger_time:
                    should_trigger = True
            elif task.task_type == "interval":
                if current_time - task.last_triggered >= task.interval_seconds:
                    should_trigger = True

            if should_trigger:
                triggered_tasks.append(task)
                task.last_triggered = current_time

                if task.task_type == "once":
                    task.is_active = False

        if triggered_tasks:
            self._save_tasks()

    for task in triggered_tasks:
        try:
            callback(task)
        except (OSError, RuntimeError) as e:
            logger.error(f"[Scheduler] 触发任务失败：{task.task_id}, error: {e}")

    return triggered_tasks
```

#### 15.5.2 备份执行

```python
def _perform_backup(self):
    """执行数据备份"""
    try:
        self.backup_dir.mkdir(parents=True, exist_ok=True)
        timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
        backup_file = self.backup_dir / f"scriptor_backup_{timestamp}.tar.gz"
        
        logger.info(f"[Scheduler] 开始执行自动备份：{backup_file}")
        
        # 排除不需要备份的目录 (如 backups 本身)
        def exclude_filter(tarinfo):
            if "backups" in tarinfo.name or "chroma_db" in tarinfo.name:
                return None
            return tarinfo

        with tarfile.open(backup_file, "w:gz") as tar:
            tar.add(self.data_dir, arcname=self.data_dir.name, filter=exclude_filter)
            
        logger.info(f"[Scheduler] 自动备份完成：{backup_file}")
        
        # 清理旧备份 (保留最近 7 天)
        self._cleanup_old_backups()
        
    except (OSError, RuntimeError) as e:
        logger.error(f"[Scheduler] 自动备份失败：{e}")
```

#### 15.5.3 旧备份清理

```python
def _cleanup_old_backups(self):
    """清理超过 7 天的旧备份"""
    try:
        now = time.time()
        retention_seconds = 7 * 24 * 3600
        
        for backup_file in self.backup_dir.glob("scriptor_backup_*.tar.gz"):
            if backup_file.is_file():
                file_age = now - backup_file.stat().st_mtime
                if file_age > retention_seconds:
                    backup_file.unlink()
                    logger.info(f"[Scheduler] 已清理过期备份：{backup_file.name}")
    except OSError as e:
        logger.error(f"[Scheduler] 清理旧备份失败：{e}")
```

在项目发展到一定阶段后，为了进一步提升系统的稳定性、性能和安全性，我们深入研究了业界领先的 AI Agent 框架（如 Claude Code 及其 Rust 重构版本），并结合本项目的实际情况，引入了多项高 ROI 的企业级架构模式。

#### 16.1 学习与借鉴的核心理念

1. **性能与成本优化**：AI Agent 的核心瓶颈往往在于 LLM 的响应时间和 Token 消耗。通过本地拦截和路由，可以大幅降低不必要的 LLM 调用。
2. **系统稳定性与容错**：工具调用是 Agent 最容易出错的环节（如幻觉循环、参数错误）。引入熔断机制（Circuit Breaker）可以有效防止系统崩溃。
3. **细粒度的安全控制**：随着工具数量的增加，必须对敏感操作（如文件删除、系统命令）进行严格的权限管控，防止恶意或意外的破坏。

#### 16.2 核心架构改进

本次升级主要引入了三个核心组件，它们通过 ArchitectureEnhancementManager 统一管理，并无缝集成到现有的 Mixin 架构中。

##### 16.2.1 Slash Command Router (斜杠命令路由器)

- **设计目标**：拦截特定的用户指令（如 /help, /status, /clear），在本地直接执行并返回结果，避免将这些简单的控制指令发送给 LLM。
- **实现机制**：
  - 采用前置拦截器模式。
  - 支持命令注册、别名和参数解析。
- **收益**：
  - **响应速度提升**：本地执行耗时在毫秒级，而 LLM 调用通常需要数秒。
  - **Token 节省**：避免了将大量上下文发送给 LLM，显著降低了 API 成本。

##### 16.2.2 Tool Health Manager (工具健康与降级管理器)

- **设计目标**：解决 LLM 在调用工具时可能出现的“幻觉循环”（即反复使用错误的参数调用同一个工具导致连续失败）。
- **实现机制**：
  - **熔断器模式 (Circuit Breaker)**：记录每个工具的调用状态。当某个工具连续失败次数达到阈值（如 3 次）时，将其状态标记为“降级（Degraded）”。
  - **动态上下文**：在向 LLM 注入工具列表时，自动过滤掉已降级的工具，防止 LLM 继续尝试。
  - **自动恢复 (半开状态探测)**：后台运行一个异步轮询任务，当工具降级超过冷却时间（如 10 分钟）后，将其临时移出降级列表（半开状态）。如果下一次调用成功，则完全恢复；如果再次失败，则重新降级并延长冷却时间。
  - **手动干预**：提供管理员命令或 API 强制恢复工具状态。
- **收益**：
  - 极大提升了系统的鲁棒性，避免了因单一工具故障导致的整个会话卡死。

##### 16.2.3 Tool Permission Engine (工具权限引擎)

- **设计目标**：为工具调用提供细粒度的安全防护，特别是针对具有破坏性的工具。
- **实现机制**：
  - **Hook 机制**：在工具执行前（PreToolUse）触发权限检查。
  - **规则引擎**：支持基于正则表达式和通配符的规则匹配。
  - **三态控制**：
    - ALLOW：直接放行。
    - DENY：直接拒绝，并向 LLM 返回权限不足的提示。
    - ASK：挂起执行，向用户发送确认请求，等待用户授权后再继续。
- **收益**：
  - 实现了企业级的安全管控，确保敏感操作（如修改核心配置、删除重要文件）必须经过人工确认。

#### 16.3 架构集成与测试

本次架构升级通过 `ArchitectureEnhancementManager` 统一管理三个核心组件，并在 `EventsMixin` 中完成与 AstrBot 框架的集成。所有组件均通过了单元测试和集成测试，确保在真实场景中的稳定性和可靠性。

### 17. 隐私保护与 Prompt Caching 架构升级 (Phase 9)

随着项目的深入发展，我们在隐私保护策略和 Prompt 构建优化方面进行了重大升级，进一步强化了"管家"人设，并引入了 LLM Prompt Caching 机制以大幅降低 Token 成本。

#### 17.1 隐私保护策略升级：管家式隐私化解

**设计背景**：
在群聊场景中，当群成员 A 试图打听群成员 B 的隐私时，传统的硬编码权限拦截（"我没有权限"）显得生硬且缺乏人情味。我们决定将隐私保护融入"管家"的人设中，让 AI 基于管家的职业素养，用高情商的方式化解隐私冲突。

**核心改进**：

1. **VFS 路径强制隔离**：
   - 所有模板文件中的路径引用统一更新为虚拟文件系统 (VFS) 路径：
     - `@personal/`：映射到当前用户的个人物理目录
     - `@group/`：映射到当前群组的物理目录
     - `@root/`：映射到系统全局根目录（受严格的 Sudo 权限保护）
   - AI 在工具调用中只能看到 VFS 路径，无法感知物理路径，从源头杜绝路径穿越攻击。

2. **管家人设的软性隐私保护**：
   - 在 `G_SOUL.md` 和 `P_SOUL.md` 中明确注入"管家"身份定位：
     > "你是大家的管家，要保守每个人的秘密。当有人打听其他人的隐私时，不要生硬地拒绝，而是用高情商的方式化解。"
   - 示例对话：
     ```
     群成员 A："管家，你知道 B 的工资是多少吗？"
     AI："B 先生的私人事务我不便透露呢。不过大家可以多在群里交流工作心得呀~"
     ```
   - 这种基于 Prompt 的软性隐私保护，比硬编码的权限拦截更具人情味，也更符合"管家"的人设定位。

3. **三级隐私边界的 VFS 映射**：
   - `group` 级别：只能访问 `@group/` 目录，无法跨群访问
   - `personal` 级别：只能访问当前用户的 `@personal/` 目录
   - `cross` 级别：跨用户/跨群访问（受严格限制，需要充分理由）

**涉及文件**：
- `templates/personal/P_SOUL.md`：更新个人核心准则，强化管家身份
- `templates/group/G_SOUL.md`：添加"你正处于群聊场景中"的上下文提示，明确群组管家定位
- `templates/personal/P_AGENTS.md`：修复残留的相对路径，100% 使用 VFS 路径
- `templates/group/G_PROFILE.md`：更新群组画像，明确隐私边界

#### 17.2 Prompt Caching 极致优化：Global -> Group -> Personal 优先级策略

**设计背景**：
在将核心文件（`PROFILE`、`AGENTS`、`GROUP`）从渐进式披露改为常驻加载后，System Prompt 的 Token 数量有所增加。为了弥补这一损失并进一步优化成本，我们深入研究了 LLM 的 Prompt Caching 机制，并在 `PromptBuilder` 中实现了严格的加载优先级策略。

**Prompt Caching 原理**：
- LLM（如 Claude、GPT-4）会对 System Prompt 的前缀进行缓存。
- 如果多个请求的 System Prompt 前缀完全一致，LLM 可以直接命中缓存，无需重新计算，从而大幅降低 Token 成本和响应延迟。
- 在多用户群聊场景中，如果不同用户发言时的 System Prompt 前 70% 是完全一致的，那么这部分前缀就能完美命中缓存。

**加载优先级设计**：

```python
# core/prompt_builder.py
class PromptBuilder:
    def _build_system_prompt_with_control(self, uid: str, group_id: str, user_message: str):
        """
        构建系统提示词（严格优先级顺序）
        
        加载顺序（从前到后）：
        1. [100] 全局核心 (Global) — 所有用户/群聊共享，绝对静态
        2. [99-97] 群组核心 (Group) — 仅群聊加载，同一群内所有用户共享，相对静态
        3. [96-94] 个人核心 (Personal) — 个人专属，动态变化
        """
        
        # ========== 第一层：绝对静态区 (Static Zone) - 命中缓存的核心 ==========
        # [100] 全局核心准则 (Global SOUL.md)
        global_soul = self._load_resident_file("global/SOUL.md")
        
        # [99] 群组核心准则 (G_SOUL.md)
        if group_id:
            group_soul = self._load_resident_file(f"groups/{group_id}/G_SOUL.md")
        
        # [98] 群组公共身份与关系网 (G_PROFILE.md)
        if group_id:
            group_profile = self._load_resident_file(f"groups/{group_id}/G_PROFILE.md")
        
        # [97] 群组工作流与行为守则 (G_GROUP.md)
        if group_id:
            group_group = self._load_resident_file(f"groups/{group_id}/G_GROUP.md")
        
        # ========== 第二层：个人专属区 (Personal Zone) - 破坏缓存的变量 ==========
        # [96] 个人核心准则 (P_SOUL.md)
        personal_soul = self._load_resident_file(f"profiles/{uid}/P_SOUL.md")
        
        # [95] 个人画像与人设 (P_PROFILE.md)
        personal_profile = self._load_resident_file(f"profiles/{uid}/P_PROFILE.md")
        
        # [94] 个人行为准则与操作手册 (P_AGENTS.md)
        personal_agents = self._load_resident_file(f"profiles/{uid}/P_AGENTS.md")
```

**缓存命中分析**：

| 场景 | 用户 A 发言 | 用户 B 发言 | 缓存命中率 |
|------|----------|----------|----------|
| **旧版（ interleaved）** | Global + P_A + Group + ... | Global + P_B + Group + ... | 0% (Personal 破坏了 Group 缓存) |
| **新版（优先级）** | Global + Group + P_A + ... | Global + Group + P_B + ... | 70%+ (Global+Group 完全一致) |

**收益**：
- **Token 成本降低**：在多用户群聊中，70% 的 System Prompt 内容可以命中缓存，实际计费 Token 减少 60%+。
- **响应速度提升**：缓存命中的请求响应延迟降低 40%+。
- **扩展性增强**：支持更多用户同时在群聊中与 AI 交互，而不会导致 Token 成本线性增长。

**涉及文件**：
- `core/prompt_builder.py`：完全重构 `_build_system_prompt_with_control` 和 `_build_system_prompt_without_control` 方法，实现严格的优先级加载逻辑。
- `core/background_tasks.py`：更新 Heartbeat 系统，使用新的 VFS 文件名（`P_HEARTBEAT.md`、`G_HEARTBEAT.md`）。

#### 17.3 常驻上下文 vs 渐进式披露：平衡之道

**核心思想**：
- **核心常驻（Resident Context）**：`PROFILE` (画像)、`AGENTS/GROUP` (行为守则) 等决定 AI 基础认知和行为边界的文件，作为常驻上下文完整加载，防止 AI "盲目行动"。
- **渐进式披露（Progressive Disclosure）**：`MEMORY` (长期记忆)、`SKILL` (技能) 等大容量数据文件，采用目录索引模式。System Prompt 只包含精简的文件目录树，AI 通过 `file_read_tool` 按需读取。

**Token 优化效果对比**：

| 项目 | 旧模式 | 新模式 | 优化 |
|------|--------|--------|------|
| **基础 Prompt** | 5000+ Token | 1500-2000 Token | **60%+ 减少** |
| **PROFILE.md** | 全量加载 | 全量加载 (常驻) | **0% (保证认知)** |
| **MEMORY.md** | 全量加载 (3000 Token) | 目录索引 (80 Token) | **97% 减少** |
| **SOP.md** | 全量加载 (1500 Token) | 目录索引 (60 Token) | **96% 减少** |

**设计哲学**：
- 在保证 AI 核心认知不缺失的前提下，尽可能减少 Token 消耗。
- 通过 Prompt Caching 机制，将常驻上下文带来的 Token 增加降到最低。
- 让 AI 在"知情"的前提下行动，而不是"盲人摸象"。

## 第五部分：高级能力扩展与工程化完善 (Phase 9)

### 17. 借鉴 Claude Code 的高级能力扩展

在完成了基础的企业级架构升级后，我们继续深入挖掘 Claude Code 的优秀设计，针对多用户并发、工具自省、技能宏调用等高级场景，对 Scriptor 进行了新一轮的扩展与完善。

#### 17.1 并发控制系统 (Concurrency Guard)

- **背景与痛点**：随着用户量的增加，多用户或多群组同时触发 AI 响应时，容易导致资源竞争、上下文混乱甚至 API 速率限制（Rate Limit）报错。
- **设计与实现**：
  - 引入了 ConcurrencyGuard 组件，实现全局并发控制。
  - **优先级队列**：支持基于优先级的请求排队（如管理员指令优先、VIP 用户优先）。
  - **会话隔离锁**：确保同一个会话（Session）在同一时刻只有一个请求在处理，避免上下文读写冲突。
- **收益**：彻底解决了多并发场景下的混乱问题，保障了系统在高负载下的稳定运行。

#### 17.2 WebFetch 工具：补齐网页读取盲区

- **背景与痛点**：原有的 web_search 工具擅长通过搜索引擎获取摘要，但无法直接读取用户提供的特定 URL 的完整内容。
- **设计与实现**：
  - 新增 WebFetcher 工具，专门用于直接抓取和解析指定 URL 的网页内容。
  - 支持 HTML 到 Markdown 的智能转换，去除冗余标签，提取核心正文。
- **收益**：大幅提升了 AI 处理特定网页链接（如新闻文章、技术文档）的能力，与 web_search 形成完美互补。

#### 17.3 ToolSearch：赋予 LLM 工具自省能力

- **背景与痛点**：随着工具数量增加到几十个，LLM 容易“忘记”或“选错”工具，导致任务失败。
- **设计与实现**：
  - 新增 ToolSearchEngine，允许 LLM 主动搜索当前可用的工具列表。
  - 支持基于关键词、分类（Category）和描述（Description）的模糊搜索。
- **收益**：让 LLM 具备了“自省”能力，在遇到复杂任务时，可以先搜索合适的工具，再进行调用，显著降低了工具选择错误率。

#### 17.4 循环提醒：自然语言时间表达式解析

- **背景与痛点**：原有的定时任务系统只支持简单的绝对时间，不支持“每周一”、“每天早上 8 点”等符合人类直觉的循环表达式。
- **设计与实现**：
  - 引入 ecurrence_parser.py，支持解析复杂的自然语言时间表达式（如 "every monday", "every weekday at 9am"）。
  - 与现有的 Scheduler 深度集成，支持生成周期性任务。
- **收益**：极大提升了定时提醒功能的用户体验，使其更接近真实的人类助手。

### 18. 核心大脑升级：Skill 系统的重构与进化

Skill（技能）系统是 Scriptor 的核心特色之一。在本次升级中，我们将 Skill 从“被动文档”升级为“主动技能宏”，并对其元数据和加载机制进行了彻底重构。

#### 18.1 极简元数据与智能推荐

- **极简元数据**：废弃了过去冗长的 Markdown 技能描述，改为极简的 YAML/JSON 格式，仅保留 
ame, description, when-to-use, llowed-tools 四个核心字段。
- **智能推荐注入**：
  - 在每次对话前，系统会根据用户的输入，利用 when-to-use 字段进行语义/关键词匹配。
  - 将匹配度最高的 Top 2 技能说明，动态注入到 System Prompt 中。
- **收益**：在不增加全局 Prompt 负担的前提下，精准提升了 LLM 在特定场景下的技能使用准确率。

#### 18.2 主动技能宏调用 (SkillTool)

- **从被动到主动**：过去，Skill 只是作为 Prompt 提示 LLM“应该怎么做”。现在，新增了 SkillExecutor 和 SkillTool，允许 LLM 像调用普通工具一样，直接调用一个复杂的“技能宏”。
- **工具白名单 (Allowed Tools)**：在执行特定技能时，系统会严格按照技能定义中的 llowed-tools 过滤可用工具，防止 LLM 在执行特定任务时“节外生枝”。

#### 18.3 基础工程完善：双级加载与后台任务

- **双级加载机制**：
  - 支持从插件自带的默认目录加载内置技能。
  - 支持从用户自定义目录（如 data/scriptor_custom_skills）加载自定义技能。
  - 支持同名技能覆盖，方便用户深度定制。
- **后台任务管理**：
  - 针对耗时较长的技能（Forked 任务），新增了 skill_status_tool 和 skill_cancel_tool。

## 第六部分：虚拟文件系统与认知减负 (Phase 10)

### 19. 虚拟文件系统 (VFS) 架构升级

在项目演进过程中，我们发现 AI 在处理跨目录文件操作时，经常因为复杂的物理路径（如包含哈希值的 `profiles/user_4d3d9584/`）而产生认知负担，导致路径拼接错误或权限越界。为了彻底解决这一问题，我们引入了虚拟文件系统（Virtual File System, VFS）架构。

#### 19.1 核心设计理念

1. **向 AI 隐藏物理复杂性**：AI 不再需要知道底层的物理目录结构（如 UID 哈希、群组 ID），而是使用语义化的虚拟命名空间。
2. **扁平化统一工作区**：提供一个统一的、扁平化的虚拟工作区，降低 AI 的认知负担。
3. **所见即所得的闭环**：AI 看到的路径、请求的路径、以及工具返回的路径，全部统一为虚拟路径，形成逻辑闭环。
4. **继承并强化 RBAC 权限**：在虚拟路径解析层集成基于角色的访问控制（Role-Based Access Control），确保跨域访问的安全性。

#### 19.2 核心命名空间设计

VFS 架构定义了三个核心的虚拟命名空间前缀：

- `@personal/`：映射到当前用户的个人物理目录（如 `profiles/user_xxx/`）。AI 读写自己的文件时，只需使用 `@personal/P_PROFILE.md`。
- `@group/`：映射到当前群组的物理目录（如 `groups/group_xxx/`）。在群聊场景下，AI 读写群文件时，只需使用 `@group/G_PROFILE.md`。
- `@root/`：映射到系统的全局根目录（如 `global/` 或其他用户的目录）。**此命名空间受到严格的 Sudo 权限保护**，仅允许管理员在跨域查看或全局维护时使用。

#### 19.3 技术实现机制

1. **拦截-映射-反向格式化 (Interceptor-Mapper-Formatter)**：
   - **输入拦截**：在所有文件操作工具（如 `file_read`, `file_write`, `file_list`）的入口处，拦截 AI 传入的虚拟路径。
   - **路径解析 (`_resolve_vfs_path`)**：将虚拟路径（如 `@personal/test.txt`）解析为真实的物理绝对路径，并在此过程中进行权限校验（如检查是否具有 `@root/` 的访问权限）。
   - **反向格式化 (`_format_vfs_result`)**：在工具执行完毕返回结果给 AI 之前，将结果中的物理路径替换回对应的虚拟路径。确保 AI 看到的始终是虚拟路径。

2. **工具全面改造**：
   - `file_list`：支持虚拟根目录 `.` 展示可用的命名空间。
   - `file_read`：读取文件内容后，如果内容中包含物理路径引用，也会被格式化为虚拟路径。
   - `file_write`, `file_edit`, `file_append`：全面接入 VFS 解析器，同时保留了原有的防缩水、只读拦截等安全机制。

3. **Prompt 与模板更新 (Context Injection)**：
   - 重写了 `PromptBuilder` 中的工作区上下文构建逻辑，向 AI 详细解释 VFS 架构和可用命名空间，并提供正反例演示。
   - 更新了 `P_SOUL.md` 和 `G_SOUL.md` 等核心模板，彻底废弃了旧的物理路径说明，替换为 VFS 规则。

#### 19.4 收益与总结

通过引入 VFS 架构，Scriptor 实现了文件管理系统的重大升级：
- **极大地降低了 AI 的认知负担**：AI 不再需要处理复杂的哈希 ID 和路径拼接，只需关注语义化的命名空间，显著减少了文件操作错误。
- **提升了系统的安全性**：权限控制被下沉到 VFS 解析层，`@root/` 命名空间的引入使得跨域访问和全局维护更加规范和安全。
- **增强了架构的扩展性**：未来如果底层物理存储结构发生变化，只需修改 VFS 映射逻辑，对 AI 而言是完全透明的。

这次升级标志着 Scriptor 在“AI 友好型架构”上迈出了关键一步，让 AI 能够更加专注、高效、安全地管理用户和群组的数字资产。

### 20. Read-Before-Write 架构：文件操作的终极防线

在 VFS 架构解决了“路径认知负担”后，我们发现 AI 在修改文件时仍存在一个致命的幻觉问题：**未读取文件内容就直接盲目覆写（导致内容严重缩水）或盲目编辑（导致匹配失败）**。虽然我们之前引入了“防缩水拦截器”作为事后兜底，但这是一种“试错”机制，会白白浪费一次 LLM 调用（Token 和时间）。

为了从根源上解决这个问题，我们深度借鉴了 Claude Code 的文件操作工具设计，引入了 **Read-Before-Write（先读后写）架构**。

#### 20.1 核心设计理念

1. **事前强制约束**：将“防缩水”从**事后惩罚**升级为**事前拦截**。AI 必须先在脑海中构建完整的上下文（读取文件），才能被允许修改文件。
2. **会话级状态感知**：赋予工具“记忆”能力，系统需要知道 AI 在当前会话中是否已经读过某个文件。
3. **极致的 Token 优化**：如果 AI 重复读取一个未发生变化的文件，系统应直接拦截并提示其查阅历史记录，避免重复下发大量代码/文本。

#### 20.2 核心组件与实现

##### 20.2.1 会话级读取状态管理器 (Read State Manager)
- **机制**：在内存中维护一个全局字典 `_SESSION_READ_STATES`，以 `session_id` (如 `uid_gid`) 为键，记录 AI 在当前会话中读取过的文件信息。
- **记录内容 (`FileReadRecord`)**：
  - `timestamp`：读取时的文件物理修改时间 (mtime)，用于检测文件是否被外部篡改。
  - `is_partial`：布尔值，记录 AI 是全量读取还是局部读取（使用了 start_line/end_line）。
- **生命周期**：每次成功调用 `file_read` 时记录状态；每次成功写入/编辑后清除该文件的状态；支持过期会话的自动清理。

##### 20.2.2 强制拦截器 (Read-Before-Write Guardian)
在所有写操作工具的入口处增加前置校验：
- **`file_write` (全量覆写)**：**最严格的拦截**。如果文件已存在，必须校验其状态为“已全量读取”（`is_partial=False`）。否则直接报错拒绝执行。
- **`file_edit` / `multi_edit` (局部编辑)**：**中度拦截**。必须校验其状态为“已读取”（允许局部读取）。
- **`file_append` (追加)**：**宽松模式**。不强制拦截，但如果未读取会返回强烈的警告建议。
- **外部篡改检测**：如果写入前发现文件的当前 mtime 大于记录的 timestamp，说明文件被用户或其他程序修改过，强制要求 AI 重新读取。

##### 20.2.3 读取去重机制 (Dedup)
- **机制**：在 `file_read` 入口处，如果发现 AI 请求读取的文件已经在状态记录中，且物理 mtime 未发生变化，且请求的行范围一致。
- **动作**：直接返回一个极短的 Stub（存根）提示：*"文件未变更，请直接参考你之前的聊天记录中的文件内容进行操作。"*
- **收益**：极大地节省了长上下文场景下的 Token 消耗。

##### 20.2.4 模糊匹配容错升级 (Fuzzy Match Enhancement)
为了配合 `file_edit` 成为修改文件的首选工具，我们大幅增强了 `_fuzzy_replace` 的容错能力，引入了 **5 级降级匹配策略**：
1. **精确匹配**（完全一致）
2. **行尾空格容错**（忽略大模型经常遗漏的行尾空格/Tab）
3. **引号标准化**（自动将大模型生成的中文弯引号 `“ ”` 转换为代码中常用的英文直引号 `" "`）
4. **空白字符容错**（将连续的换行/空格视为等价）
5. **组合容错**（引号 + 行尾空格 + 空白字符）

#### 20.3 Prompt 深度协同

纯代码拦截是不够的，必须在 AI 规划任务时就植入规则。我们在系统提示词（System Prompt）中进行了深度协同：
- 在 **日常操作指南** 中新增硬性规定：“在修改任何已有文件之前，必须先使用 file_read_tool 读取该文件，否则操作将被系统拦截。”
- 在 **反模式（避免的错误用法）** 中明确指出“不要盲目覆写文件”。
- 在 **工具描述** 中，强烈建议将 `file_write` 仅用于创建新文件，而将 `file_edit` 作为修改已有文件的首选工具。

#### 20.4 收益与总结

Read-Before-Write 架构的引入，标志着 Scriptor 的文件操作工具达到了工业级水准：
- **消灭了盲目试错**：AI 不再会因为没读文件而触发防缩水拦截，节省了大量无效的 API 调用。
- **大幅降低了成本**：Dedup 机制有效防止了 AI 的“健忘性重复读取”。
- **提升了编辑成功率**：5 级模糊匹配容错让 `file_edit` 变得极其好用，AI 不再因为一个全角引号或行尾空格而反复修改失败。

### 22. 中文分词优化 - jieba 分词库引入

#### 22.1 问题分析

原有的 BM25 分词实现使用简单的正则表达式：

```python
# 原实现（逐字分词）
def tokenize_for_bm25(text: str) -> List[str]:
    return re.findall(r"[a-zA-Z0-9]+|[\u4e00-\u9fa5]", text.lower())

# 效果
"我爱北京天安门" → ['我', '爱', '北', '京', '天', '安', '门']
"人工智能很有趣" → ['人', '工', '智', '能', '很', '有', '趣']
```

**问题**：中文逐字分词导致搜索精确度低，无法识别有意义的词汇。

#### 22.2 解决方案

引入 jieba 分词库，实现按词分词：

```python
# 新实现（jieba 分词）
def tokenize_for_bm25(text: str) -> List[str]:
    if not text:
        return []
    
    if JIEBA_AVAILABLE:
        # 使用 jieba 进行中文分词
        # cut_for_search 模式适合搜索引擎，会对长词再切分
        tokens = list(jieba.cut_for_search(text.lower()))
        return [t.strip() for t in tokens if t.strip()]
    else:
        # 回退到简单的正则分词
        return re.findall(r"[a-zA-Z0-9]+|[\u4e00-\u9fa5]", text.lower())

# 效果
"我爱北京天安门" → ['我', '爱', '北京', '天安', '天安门']
"人工智能很有趣" → ['人工', '智能', '人工智能', '很', '有趣']
```
