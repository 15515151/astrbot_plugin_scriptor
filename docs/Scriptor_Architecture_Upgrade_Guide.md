# Scriptor 架构升级指南

从传统记忆系统升级到 Scriptor 架构的对比分析。

---

## 一、存储方式对比

### 传统方案：封闭数据库

```
对话 → 写入数据库 → SQL 查询
```

**问题**：
- 数据存储在封闭的数据库中
- 用户无法直接查看和编辑记忆
- 备份和迁移复杂
- 不支持版本控制

### Scriptor：文件即记忆

```
对话 → 写入 Markdown 文件 → 文件系统
```

**优势**：

**透明性**：
- 所有记忆以 Markdown 格式存储
- 用户可以直接用文本编辑器查看和编辑
- 记忆格式人类可读，机器可解析

**可控性**：
- 记忆完全属于用户
- 可以自由备份、迁移、分享
- 支持 Git 版本控制

**扩展性**：
- 基于文件系统的架构便于扩展
- 可以与其他工具集成
- 无需特殊数据库管理

**示例对比**：
```
传统方案：
记忆存储在 SQLite/PostgreSQL 中
需要 SQL 查询才能访问
用户无法直接编辑

Scriptor：
记忆存储在 profiles/{uid}/MEMORY.md
直接用 VS Code 打开即可编辑
Git 自动记录变更历史
```

## 二、身份管理对比

### 传统方案：单平台身份

```
平台A: ID_123
平台B: ID_456
平台C: ID_789
（三个独立的身份，记忆不互通）
```

**问题**：
- 每个平台身份独立
- 记忆无法跨平台共享
- 用户体验不一致

### Scriptor：跨平台身份聚合

```
物理ID:
- 平台A: ID_123
- 平台B: ID_456
- 平台C: ID_789
    ↓ 绑定
逻辑UID: user_abc
（所有平台共享同一个身份和记忆）
```

**优势**：

**身份统一**：
- 不同平台的身份绑定到同一个逻辑 UID
- 记忆跨平台共享
- 用户体验一致

**绑定机制**：
```
1. 在设备 A 上发送 /whoami，获取绑定码
2. 在设备 B 上发送 /bind <绑定码>
3. 两个设备的身份自动合并
```

**数据管理更清晰**：
```
profiles/
├── user_abc/
│   ├── PROFILE.md
│   ├── MEMORY.md
│   └── memory/
│       ├── 2026-03-10.md
│       └── 2026-03-11.md
└── user_def/
    └── ...
```

## 三、检索策略对比

### 传统方案：单一检索

```
查询 → 单一检索方法 → 返回结果
```

**问题一：检索不精确**
- 关键词搜索无法理解语义
- 向量搜索无法精确匹配关键词
- 无法兼顾精确性和语义理解

**问题二：结果质量不稳定**
- 新记忆和旧记忆混在一起
- 重要记忆和临时记忆没有区分
- 没有时间衰减机制

### Scriptor：混合检索 + Entity-First

**检索策略：Entity-First**

```
查询 → Entity-First 排序
     → BM25 关键词检索
     → 向量语义检索
     → 结果合并与去重
     → 时间衰减排序
     → 返回最终结果
```

**优势**：

**Entity-First 排序**：
- 优先检索个人画像（核心知识）
- 其次检索标准流程（行动指南）
- 然后检索长期记忆（事实、经验）
- 最后检索近期日记和群体记忆

**混合检索**：
- BM25：精确关键词匹配，速度快
- 向量搜索：语义相似性匹配，理解上下文
- 两者结合，兼顾精确性和语义理解

**时间衰减**：
```
权重 = useful_score × (1 - 衰减系数 × 时间)
T0 易逝档：衰减快，适合临时信息
T1 待证档：衰减适中，需要验证
T2 永存档：衰减极慢，核心知识
```

**检索精度提升**：
| 场景 | 传统方案 | Scriptor |
|------|---------|----------|
| 查询"苹果" | 混合返回所有结果 | Entity-First + 混合检索 |
| 查询"Python" | 可能返回无关内容 | 精确到个人画像和核心知识 |

## 四、记忆巩固对比

### 传统方案：无记忆巩固

```
传统记忆系统 = 静态存储
```

**缺失**：
- 无法自动整理记忆
- 无法识别重复信息
- 无法发现模式和规律
- 记忆越来越混乱

### Scriptor：睡眠巩固机制

```
Scriptor = 主动记忆管理 + 睡眠巩固
```

**睡眠巩固做什么**：

| 能力 | 说明 |
|------|------|
| 模式识别 | 发现重复出现的模式和规律 |
| 目标发现 | 识别潜在的长期目标 |
| 冲突解决 | 合并矛盾的信息 |
| 冷热分离 | 将过期记忆归档 |
| 记忆合并 | 合并相似的记忆 |

**睡眠巩固流程**：
```
触发条件：新会话（跨天且超过60分钟未活跃）
    ↓
第一步：模式识别
    - 发现重复模式
    - 识别长期目标
    ↓
第二步：冲突解决
    - 检测矛盾信息
    - 合并相似记忆
    ↓
第三步：冷热分离
    - [ARCHIVE]：归档记忆
    - [MERGE]：待合并记忆
    - 活跃记忆保持
    ↓
第四步：记忆合并
    - 合并相似记忆
    - 提升记忆质量
    - 减少冗余
```

**记忆巩固 vs 不巩固的区别**：

| 维度 | 无巩固 | 睡眠巩固 |
|------|--------|---------|
| 记忆质量 | 越来越混乱 | 持续优化 |
| 冗余程度 | 大量重复 | 自动合并 |
| 检索效率 | 越来越低 | 保持高效 |
| 用户体验 | 越来越差 | 越来越好 |

## 五、群体记忆对比

### 传统方案：无群体记忆

```
传统记忆系统 = 仅个人记忆
```

**缺失**：
- 无法管理群聊记忆
- 无法跨群传递信息
- 无法支持群体协作

### Scriptor：个人 + 群体双轨记忆

```
Scriptor = 个人记忆 + 群体记忆 + 跨群传递
```

**群体记忆系统**：

| 能力 | 说明 |
|------|------|
| 群体记忆 | 管理群聊的集体记忆 |
| 成员管理 | 跟踪群体成员和角色 |
| 跨群传递 | 在不同群体之间传递信息 |
| 跨群提醒 | 创建跨群提醒任务 |

**数据结构**：
```
groups/
├── group_abc/
│   ├── GROUP.md       # 群体信息
│   ├── MEMORY.md      # 群体记忆
│   ├── SOP.md         # 群体流程（可选）
│   └── memory/        # 群体日记
│       └── 2026-03-10.md
└── group_def/
    └── ...
```

**跨群信息传递**：
```
用户在群 A 说："别忘了明天的会议"
    ↓
系统检测到跨群信息
    ↓
创建跨群提醒任务
    ↓
自动投递到用户参与的其他群
```

## 总结对比

| 维度 | 传统记忆系统 | Scriptor |
|------|-------------|----------|
| **存储方式** | 封闭数据库 | 文件即记忆（Markdown） |
| **身份管理** | 单平台身份 | 跨平台身份聚合 |
| **检索策略** | 单一检索 | Entity-First + 混合检索 |
| **记忆巩固** | 无 | 睡眠巩固机制 |
| **群体记忆** | 无 | 个人 + 群体双轨 |
| **透明性** | 低 | 高（直接编辑） |
| **可控性** | 低 | 高（完全属于用户） |
| **扩展性** | 低 | 高（文件系统） |

**升级价值**：
- 透明可控：记忆完全属于用户，可直接编辑
- 身份统一：跨平台身份聚合，体验一致
- 检索精准：Entity-First + 混合检索
- 智能巩固：睡眠持续优化记忆质量
- 群体协作：个人 + 群体双轨记忆
